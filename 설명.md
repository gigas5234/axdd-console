# AXDD SkillOps Console — 설명서

> 자연어 한 줄로 디자이너·기획자·개발자가 함께 쓸 산출물을 만들고, 그걸 Claude Code에 그대로 가져다 쓸 수 있게 묶어주는 운영 콘솔.

---

## 0. 한 줄 요약

**"신규 헬스케어 SaaS 환자 대시보드 기획해줘"** 한 줄 입력 →
도메인 인식 → 5단계 스킬 체인 자동 실행 → 디자이너용 마스터 핸드오프 문서 + Figma 프롬프트 →
**Claude Code 호환 zip 패키지**로 다운로드.

## 1. 무엇을 풀려고 만들었나

AXDD 프로젝트의 스킬·에이전트·하네스 룰이 코드와 문서에 흩어져 있어서,
**기획자·UX 디자이너가 보기에 실체가 명확하지 않다**는 문제가 출발점이었다.

이 콘솔이 해결하는 것:

1. **개념을 화면으로 보이게** — 8개 스킬 카테고리, 워크유닛, Hook이 카드와 다이어그램으로
2. **자연어 → 산출물 자동 생성** — Sandbox에서 한 줄 입력으로 디자이너 향 산출물 생성
3. **도메인 일관성 보장** — 사용자가 요청한 도메인을 모든 산출물에 일관 유지 (가장 큰 차별점)
4. **Claude Code로 외부 전달** — 산출물 + 스킬셋을 표준 zip으로 묶어 어디서든 재실행 가능

## 2. 핵심 컨셉 4가지

### 2.1 Skill — 재사용 가능한 작업 능력 (8개 카테고리)

| 카테고리 | 역할 | 예 |
|---|---|---|
| Simple | 단순 텍스트 변환 | simple-summary-skill |
| Reference | 외부 자료 참조 후 생성 | design-system-reference-skill |
| Template | 정형 템플릿 채우기 | kickoff-report-template-skill |
| Script | 외부 스크립트 호출 | html-milestone-generator-skill |
| Asset | 작업 단위 풀셋 | ux-process-asset-skill |
| Full-step | 위 4종 통합 | ux-ui-handoff-fullstep-skill |
| Metadata | 메타데이터 검색 | asset-metadata-search-skill |
| Test | 산출물 검증 | output-validation-skill |

각 스킬은 폴더 하나 (`skills/<category>/<skill-id>/`)이고 안에 `SKILL.md` · `prompt.ts` · `runner.ts` · `mock-output.ts`가 있다.

### 2.2 Work Unit — 여러 스킬의 체인

업무 1건이 보통 여러 스킬의 연쇄 실행으로 끝난다.
예) UX/UI Planning Work Unit = `simple-summary` → `design-system-reference` → `ux-process-asset` → `ux-ui-handoff-fullstep` → `output-validation`

데이터는 `data/work-units.json`에 등록되어 있고, Claude Code 배포 시에는 **단일 스킬로 패키징**된다.

### 2.3 Hook — 자연어를 워크유닛으로 라우팅

사용자가 "디자인해줘" 같은 자연어를 입력하면 어떤 워크유닛을 돌릴지 결정.

3단계 fallback:
1. **키워드 매칭** — `data/hooks.json`의 키워드와 직접 비교
2. **intent.scope** — 작업 신호(`needsHandoff`, `needsIA` 등)로 매칭
3. **intent.domain** — 도메인만 있어도 UX/UI Planning에 기본 매칭

이 덕분에 "엔터프라이즈 어드민 만들어줘" 같이 명시적 hook 키워드 없어도 동작한다.

### 2.4 Domain Profile — 도메인 일관성의 핵심

5개 도메인이 정의되어 있다. 각 프로필은 **그 도메인에 맞는 디자인 토큰·페르소나·플로우·컴포넌트**를 모두 담고 있다.

| 도메인 | 톤 | 색 토큰 | 페르소나 예 |
|---|---|---|---|
| 헬스케어 | 신뢰·차분 | teal-600 + sky-500 | 환자·보호자·의료진 |
| 핀테크 | 전문성·정확 | navy-900 + gold-500 | 송금자·자산관리·KYC |
| 이커머스 | 트렌디·활발 | coral-500 + pink-500 | MZ쇼퍼·비교형·셀러 |
| 어드민 | 효율·차분 | blue-600 + slate-50 | 분석가·운영자·관리자 |
| SaaS | 미니멀 | indigo-500 + cyan-500 | 팀리드·파워유저·신규 |

사용자가 "헬스케어"라고만 말해도 시스템이 자동으로 해당 프로필을 로드해 토큰·페르소나·화면·Figma 프롬프트가 모두 그 도메인에 맞춰 생성된다.

## 3. 콘솔 사용 흐름 — Sandbox 중심

### Step 1. 프리셋 선택 또는 자연어 입력

`/sandbox` 진입 시 상단에 9개 프리셋 카드:
- 🩺 헬스케어 SaaS 환자 대시보드 (추천)
- 🛍️ 패션 이커머스 모바일 리디자인 (추천)
- 💳 핀테크 신규 사용자 온보딩 (추천)
- 🗂️ 사내 어드민 리디자인
- 📋 신규 프로젝트 착수보고서
- 🚀 Vercel CI/CD 셋업
- ❓ 모호한 입력 (Edge case 시연용)
- 🧩 정보 일부 누락 (Edge case)
- 🤷 관련 없는 프롬프트 (Edge case)

또는 좌측 텍스트 영역에 자유롭게 입력.

### Step 2. Intent 분석 (자동, 200ms debounce)

프롬프트를 입력하는 즉시 우측에 **Intent 분석 카드** 등장.

- **Domain**: 헬스케어 / 핀테크 / 이커머스 / 어드민 / SaaS / unknown
- **ProductType**: saas / mobile-app / web / desktop / admin / unknown *(도메인과 별개로 분류)*
- **Tone**: 엔터프라이즈 / MZ / 미니멀 / 차분 / 활발 / 전문성 / unknown
- **감지된 작업 범위**: 8개 scope flag (`needsIA`, `needsHandoff` 등)
- **누락 정보**: domain / tone / timeline / team-size / target-persona / platform / 등

신뢰도(confidence)가 70% 미만이면 **Clarifying 카드**가 추가로 등장.

### Step 3. Clarifying — 정보 부족 시 자동 질문

도메인이 unknown이거나 confidence가 낮으면 노란 카드로 질문 표시.

- 어떤 도메인의 제품인가요? (필수)
- 어떤 톤·무드를 원하세요?
- 어떤 플랫폼 우선인가요?

옵션 칩을 클릭해 선택 → `[적용하고 자동 실행]` 버튼 누르면 프롬프트에 자동 합쳐지고 즉시 실행 진입.

### Step 4. 실행 — 5단계 시각화

`[실행]` 버튼 누르면 우측에서 5개 스킬이 차례로 활성화된다.

- **실행 중인 스킬**: cyan 네온 글로우 + 회전 스피너
- **완료된 스킬**: 초록 ring + 체크 아이콘
- **아직 안 돌아간 스킬**: 회색 반투명

각 스킬에 1.2~2.2초 가짜 지연이 있어 실제 LLM 호출처럼 자연스럽다.
*(이 지연은 LLM 연결 후 자동 제거되어야 함 — 코드에 명시적 주석 있음)*

실행 로그에 단계가 차례로 표시:
1. Hook matched (~500ms)
2. Work Unit selected (~600ms)
3. Skills loaded (~500ms)
4. **Output generated** — 각 스킬 1.2~2.2초씩 실행 (가장 김)
5. Validation completed (~1000ms)
6. Human review pending (~300ms)

### Step 5. 산출물 확인

실행 완료 후 우측에 산출물 카드 등장:

- **Output Preview** — 마스터 마크다운 (10개 섹션 풀세트)
  - Project Overview / IA / User Flow / Design Tokens / Component Spec / Sample Screens / Interaction & Motion / Accessibility / QA Matrix / Figma AI Prompt
  - 표·코드블록·ASCII 와이어프레임 모두 정확히 렌더링
- **Validation Result** — 4-state status + 도메인 fit 검증
  - `passed` / `passed-with-review` / `needs-review` / `failed`
  - 도메인 키워드 등장 횟수, 다른 도메인 누출 횟수 자동 측정
- **Figma 듀얼 트랙 패널**:
  - **MCP 직접 연결** (보안정책 차단 시 비활성)
  - **Figma AI 프롬프트 복사** (항상 동작, 클립보드 자동 복사)

### Step 6. 다운로드 — 두 가지 zip

#### 옵션 A. **결과만** (회색 버튼)
- 산출물·검증·intent·Figma 프롬프트만 묶은 가벼운 zip
- 디자이너·PM에게 슬랙·메일로 공유용

#### 옵션 B. **Bundle (결과 + 스킬셋)** (인디고 버튼)
- **Anthropic Skills 표준 구조** — Claude Code에 그대로 풀어 재실행 가능
- 압축 풀어 `~/.claude/skills/`에 두면 자연어로 호출 가능

Bundle 구조:
```
ux-ui-planning-workunit/        ← 폴더명 = Claude Code skill name
├── SKILL.md                    ← frontmatter (name, description, ...)
├── CATALOG.md                  ← 파일·sub-skill 인덱스
├── README.md                   ← 사람용 시작 문서
├── work-unit.json
├── references/                 ← 통합 참고자료
├── scripts/                    ← 실행 스크립트
├── assets/                     ← 산출물 템플릿
├── tests/                      ← 검증 룰
├── examples/                   ← 도메인별 샘플 4종 (Claude 학습용)
├── result/                     ← 이번 실행 산출물
└── _dev/                       ← (옵션) Next.js TS 구현 참조
```

## 4. 페이지별 기능 — 7개 탭

### 4.1 Overview (`/`)
- KPI 카드 (총 스킬·워크유닛·리뷰 대기·릴리즈 후보) + 트렌드 indicator
- **Architecture Status** — 5개 시스템 컴포넌트 (Intent 추출 / Hook 라우팅 / Domain Profiles / Validation / Bundle Export)
- Quick Actions 4개 (Sandbox / Skills / Work Units / Governance 바로가기)
- 8단계 실행 흐름 시각화 (양끝 정렬 반응형)
- Recent Runs · Activity Feed · Integration Status (3-column)
- Skill 카테고리 미니 바차트

### 4.2 Skill Library (`/skills`)
- 4개 통계 카드 (Total / Verified / Unverified / Owners)
- 8개 카테고리 필터 + 검색
- 스킬 카드 (Skills.sh 스타일)
- 카드 클릭 → 우측 슬라이드 패널:
  - Input/Output Schema
  - **SKILL.md 본문 미리보기** (Claude Code 호환 frontmatter 포함)
  - Files (references/scripts/assets/tests)
  - Related Work Units
  - **Copy / Download .md / Download zip** 버튼
  - "Claude Code에서 직접 테스트하는 법" 안내

### 4.3 Work Units (`/work-units`)
- 좌측 사이드바: 워크유닛 카드 (트리거 키워드 칩 미리보기)
- 우측: 선택된 워크유닛 상세
  - Trigger Hook 카드 (키워드 전체 노출)
  - React Flow 기반 스킬 체인 다이어그램
  - Input / Output 색상 분리 카드
  - Validation Skill 인라인
  - **`📦 Bundle zip`** 버튼 (스킬셋만)
  - **`Sandbox에서 시연`** 버튼 (deep link)

### 4.4 Sandbox (`/sandbox`) ⭐ 핵심
위 §3에서 자세히 설명.

### 4.5 Asset Repository (`/assets`)
- 4개 통계 카드 (Total / Active / Migration Candidates / High Risk)
- 검색 + 8개 탭 필터 (All / Inbox / References / Templates / Scripts / Outputs / Duplicates / Migration)
- 자산 카드 → 클릭 시 Detail Panel (우측 슬라이드)
  - 카테고리·중복 위험·Source
  - Related Skills
  - Migration / High Risk 경고 배너
  - Preview / Download 액션

### 4.6 Governance (`/governance`)
- 4개 통계 카드 (Pending / Approved / Released / Open Risks)
- **Release Pipeline** — 5단계 (Draft → Tested → Needs Review → Approved → Released) 워크유닛 분포
- **Domain Fit Distribution** — 최근 실행의 도메인 보존 통계 표
  - Run · Domain · Prompt · Hits · Leak · Status (4-state 색 코딩)
  - 하단에 Leak 범례 + Status 4-state 범례
- Review Queue — Approve/Reject/Defer 액션 버튼
- Risk Log · Decision Log

### 4.7 Docs (`/docs`)
6개 문서 카드 — 클릭 시 우측 슬라이드 패널에 풀 마크다운 렌더링.

1. 🏛 **AXDD Architecture** — 시스템 전체 가이드
2. 🚀 **Claude Code Install Guide** — Bundle을 Claude Code에 설치하는 절차
3. 📦 **Standard Kit Spec** — Anthropic Skills 표준 + AXDD 확장
4. 🧠 **Agent Creation Guide** — 새 스킬·워크유닛·Hook 추가 절차
5. 📖 **Playbook** — 3개 시나리오 + Edge-case + Bundle 체크리스트
6. 📅 **WBS** — Phase 1~5 진행 + Phase 6 (Production) 항목

## 6. 도메인 보존 시스템 — 가장 큰 차별점

기존 LLM 워크플로의 가장 큰 문제는 **사용자가 요청한 도메인을 LLM이 자기 학습 데이터로 덮어쓰는 것**이다.
예) 헬스케어 요청 → 결과물에 AXDD 콘솔 컴포넌트가 박혀 나옴.

이 콘솔이 그 문제를 잡는 방법:

### 6.1 중앙 룰
`skills/_runtime/system-rules.ts`의 `DOMAIN_PRESERVATION_RULE` 상수 — 모든 스킬의 system prompt 앞에 자동 prepend.

규칙 내용:
- 사용자 요청 도메인을 모든 산출물에 유지
- 내부 카탈로그 예시로 도메인을 덮어쓰지 말 것
- 도메인 미지정 시 일반론 + "확인 필요" 메모

### 6.2 Intent 분리
"헬스케어 SaaS"는 `domain="헬스케어"` + `productType="saas"`로 분해.
이렇게 분리해야 SaaS 같은 product type 키워드가 도메인 누출로 잘못 잡히지 않는다.

### 6.3 의미 검증 (validation skill)
산출물 본문에서 자동 측정:
- 사용자 요청 도메인 키워드 등장 횟수 (목표 ≥ 5회)
- 다른 도메인 키워드 누출 횟수 (임계 ≤ 3회)
- 4-state status: passed / passed-with-review / needs-review / failed

### 6.4 실측 결과
| 시나리오 | 도메인 키워드 | 다른 도메인 누출 |
|---|---|---|
| 헬스케어 SaaS 환자 대시보드 | 107회 | 0회 |
| 패션 이커머스 MZ | 119회 | 2회 |
| 핀테크 KYC + 송금 | 113회 | 2회 |
| 엔터프라이즈 어드민 | 66회 | 1회 |

## 7. 현재 한계점 — 솔직하게

### 7.1 LLM이 아직 안 붙어 있음
모든 실행은 **mock 모드**다. 각 스킬의 `mock-output.ts`가 도메인 프로필 기반으로 정적 텍스트를 생성한다.

- 같은 도메인 입력엔 매번 같은 결과가 나옴 (다양성 없음)
- LLM 미연결이라 실제 의도 분류·창의적 변형 없음
- 단, **인프라는 완벽히 준비됨** — `ANTHROPIC_API_KEY` 환경변수 추가만 하면 자동 LLM 전환

### 7.2 UX/UI 워크유닛 중심
현재 산출물 품질이 가장 높은 영역은 **UX/UI Planning Work Unit**이다.
- 마스터 핸드오프 (10섹션) 풀세트
- 4개 도메인 examples 자동 생성
- Figma 프롬프트 별도 추출

다른 워크유닛 (Kickoff Report, CI/CD Setup) 은 mock이 더 단순하다.

### 7.3 데이터가 영속화 안 됨
모든 데이터는 `data/*.json` + 메모리 상태.
- 페이지 새로고침 시 실행 결과 사라짐
- 자산 추가/리뷰 승인이 저장 안 됨
- Sandbox 실행 기록도 누적 안 됨

### 7.4 인증·권한 없음
누구나 접근 가능. CLAUDE.md §3에 3개 역할(Designer / Developer / Operator)이 정의돼 있지만 코드에는 아직 없음.

### 7.5 Figma MCP 미연결
사내 보안정책으로 차단된 상태로 가정.
- AI 프롬프트 추출 모드는 항상 동작 (디자이너가 Figma AI에 붙여넣기)
- MCP 직접 연결 어댑터는 슬롯만 마련됨 (`lib/figma/mcp-adapter.ts`)

## 8. 추후 작업 — Phase 6 Production

### 8.1 LLM 연결 (1순위)
\```bash
# .env.local 생성
ANTHROPIC_API_KEY=sk-ant-...
\```

이 한 줄만으로 다음이 자동 활성화:
- Intent 추출 → 휴리스틱에서 LLM 분류로
- 각 스킬 실행 → mock에서 Claude 호출로
- Validation → 정적 룰 + LLM 메타 평가

코드 변경 0. `withLlmOrMock()` 헬퍼가 자동 분기한다.

### 8.2 데이터 저장소 (2순위)
\```
data/*.json → Supabase (Postgres) / Vercel Postgres
\```

`lib/data.ts`만 fetch 기반으로 교체하면 컴포넌트는 그대로.

### 8.3 인증 (3순위)
NextAuth.js + 3개 역할:
- **Designer** — Skill Library / Sandbox / Asset 읽기·실행
- **Developer** — 위 + 스킬 추가·편집
- **Operator** — 위 + Governance (승인·릴리즈)

### 8.4 작업 큐 (LLM 도입 후)
긴 스킬 체인은 30초~수 분 걸릴 수 있어 HTTP 요청 안에서 못 끝낸다.
- Inngest / Trigger.dev / Upstash QStash
- 진행 상황은 SSE 또는 WebSocket으로 푸시

### 8.5 파일 스토리지
산출물 영구 저장:
- Vercel Blob / S3 / Cloudflare R2 / Supabase Storage

### 8.6 다른 도메인 확장
현재 5개 도메인 (헬스케어·핀테크·이커머스·어드민·SaaS).
다음 우선순위:
- 교육 (LMS·MOOC)
- 엔터테인먼트 (스트리밍·게임·미디어)
- 부동산·자동차·여행 같은 vertical SaaS

각 도메인 추가는 `skills/_runtime/domain-profiles.ts`에 새 프로필 객체 하나 추가하면 끝.

### 8.7 GitHub 연동
- 스킬 정의를 GitHub repo로 단방향 동기화 (read-only가 안전)
- `gh` API 또는 Octokit

### 8.8 Figma MCP 활성화
사내 보안정책 해제되면 `lib/figma/mcp-adapter.ts`의 `_transport`에 실제 MCP 클라이언트 wiring.
UI 변경 0 — 어댑터 인터페이스 그대로.

### 8.9 관찰성 (Observability)
- 실행 로그 (어떤 프롬프트 → 어떤 스킬 → 비용)
- Sentry (에러 추적)
- Langfuse / Helicone (LLM trace + 비용)
- PostHog (사용자 행동 분석)

### 8.10 테스트 (CI 회귀 방지)

- 어댑터 계층 (Figma, LLM) 우선
- Golden test set — 알려진 입출력 쌍으로 회귀 자동 감지

## 9. 다른 작업자가 추가 작업할 때

각 영역이 독립적이라 동시 작업 가능.

| 영역 | 진입점 |
|---|---|
| 새 도메인 추가 | `skills/_runtime/domain-profiles.ts`에 새 프로필 객체 추가 |
| 새 스킬 추가 | `skills/<category>/<id>/` 폴더 + `data/skills.catalog.json` 등록 |
| 새 워크유닛 추가 | `data/work-units.json` + (옵션) `data/hooks.json` |
| LLM 프롬프트 튜닝 | 각 스킬의 `prompt.ts` |
| Mock 결과 개선 | 각 스킬의 `mock-output.ts` |
| 검증 룰 추가 | `mocks/validation.ts` |
| UI 컴포넌트 추가 | `components/ui/` |
| 새 페이지 추가 | `app/<route>/page.tsx` + 사이드바 nav 추가 |

**가장 중요한 진입점**: `skills/_runtime/`의 8개 파일이 시스템의 두뇌다. 여기를 이해하면 나머지는 따라온다.

- `intent.ts` — 자연어 → 의도 객체
- `clarifying.ts` — 정보 부족 시 질문 카탈로그
- `domain-profiles.ts` — 5개 도메인 프로필 + countDomainKeywords
- `system-rules.ts` — DOMAIN_PRESERVATION_RULE
- `helpers.ts` — withLlmOrMock 표준 헬퍼
- `llm-client.ts` — Anthropic API 클라이언트
- `registry.ts` — 스킬 등록부
- `types.ts` — 공통 타입
